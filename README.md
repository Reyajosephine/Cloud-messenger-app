# Cloud Messenger App

## 1. Project Overview
Cloud Messenger is a one-way private messenger app, similar to WhatsApp "Message Yourself".

- Each user signs in with email/password.
- Supabase sends an email verification link for new accounts.
- After login, users can send messages to their own private chat space.
- Messages are shown with sender name, text, and timestamp.
- Frontend refreshes messages every 2 seconds (polling).

This project is intentionally simple:
- No multi-user chat rooms
- No complex role system
- No framework on frontend

The goal is to learn a full cloud flow with authentication, API security, and deployment.

## 4. Architecture

1. User opens frontend hosted on Azure Static Web Apps.
2. Frontend uses Supabase Auth for sign up/login.
3. New user receives email verification link from Supabase.
4. After login, frontend gets access token from Supabase session.
5. Frontend calls backend API on Vercel with `Authorization: Bearer <token>`.
6. Backend validates token with Supabase and reads/writes messages.
7. Frontend polls `GET /messages` every 2 seconds and updates chat UI.

Why the backend validates token:
- Frontend can never be trusted alone.
- Backend must verify user identity before returning/storing data.
- This is what ensures one user cannot read another user's messages.

## 6. Supabase Setup (Auth + Data)

### Step 1: Create Supabase project
1. Go to https://supabase.com
2. Create a new project.
3. Copy these values from Project Settings -> API:
	 - `Project URL`
	 - `anon public key`
	 - `service_role key` (backend only)

Important:
- `anon public key` is safe for frontend.
- `service_role key` is secret and must stay in backend env only.

### Step 2: Enable Email auth
1. Open Authentication -> Providers -> Email.
2. Enable Email provider.
3. Turn on email confirmation (recommended).

If you are testing quickly and not receiving emails:
1. You can disable confirmation temporarily.
2. Re-enable it before production use.

### Step 3: Create messages table
Run this SQL in Supabase SQL Editor:

```sql
create table if not exists public.messages (
	id bigint generated by default as identity primary key,
	user_id uuid not null references auth.users (id) on delete cascade,
	username text not null,
	text text not null,
	created_at timestamptz not null default now()
);

create index if not exists idx_messages_user_created
	on public.messages (user_id, created_at);

alter table public.messages enable row level security;

create policy "Users read own messages"
on public.messages
for select
using (auth.uid() = user_id);

create policy "Users insert own messages"
on public.messages
for insert
with check (auth.uid() = user_id);
```

What this SQL does:
1. Creates `messages` table.
2. Links message owner to auth user (`user_id`).
3. Enables Row Level Security (RLS).
4. Allows users to read/insert only their own rows.

## 7. Backend Setup (Node + Express on Vercel)

### Step 0: Backend repository structure
Backend repo should contain at least:
1. `api/index.js`
2. `package.json`
3. `vercel.json`

### Step 1: Configure environment variables
In backend `.env` (local) and in Vercel project settings (production), add:

```env
SUPABASE_URL=your_supabase_project_url
SUPABASE_SERVICE_ROLE_KEY=your_supabase_service_role_key
```

Why both are needed:
1. Backend validates auth tokens against Supabase.
2. Backend accesses `messages` table securely.

### Step 2: Install dependencies
From `backend` folder:

```bash
npm install
```

### Step 3: Run backend locally (optional)

```bash
npm run dev
```

Quick local checks:
1. `GET /` should return healthy status JSON.
2. `GET /messages` without token should return 401.

This confirms auth protection is active.

### Step 4: Deploy backend to Vercel
1. Push backend folder to a separate GitHub repository.
2. Import that repo in Vercel.
3. Add environment variables in Vercel:
	 - `SUPABASE_URL`
	 - `SUPABASE_SERVICE_ROLE_KEY`
4. Deploy.
5. Save the deployed API URL, for example:
	 - `https://your-backend-name.vercel.app`

After deployment:
1. Open Vercel project -> Settings -> Environment Variables.
2. Confirm both env vars are set for Production.
3. Redeploy once after env updates.

## 8. Frontend Setup (Azure Static Web Apps)

### Step 1: Update frontend config
In `frontend/script.js`, set:

```js
const API_URL = "https://your-backend-name.vercel.app";
const SUPABASE_URL = "https://your-project-ref.supabase.co";
const SUPABASE_ANON_KEY = "your-anon-key";
```

Common mistake:
- Do not put a trailing slash in `API_URL`.
- Example good: `https://your-backend.vercel.app`
- Example risky: `https://your-backend.vercel.app/`

### Step 2: Push frontend code to GitHub
Use a separate frontend repository (recommended) or keep frontend under `/frontend` in the same repo.

If your frontend is in a subfolder, make sure Azure app location matches it.

### Step 3: Add Azure Static Web Apps workflow
This project already includes:

- `.github/workflows/azure-static-web-apps.yml`

Make sure GitHub secret is added:

- `AZURE_STATIC_WEB_APPS_API_TOKEN`

If this secret is missing, GitHub workflow will fail at deploy step.

### Step 4: Create Azure Static Web App resource
1. In Azure Portal, create a Static Web App.
2. Connect your GitHub frontend repository.
3. Build config for this project layout:
	 - App location: `/frontend`
	 - Output location: `.`
4. Complete setup and wait for deployment.

When deployment finishes:
1. Open your Azure Static Web App URL.
2. Confirm page loads and login buttons are visible.

## 9. End-to-End User Flow

1. User opens the frontend URL.
2. User clicks Create Account.
3. Supabase sends verification email.
4. User verifies email and logs in.
5. User enters display name and sends message.
6. Frontend sends message to backend with bearer token.
7. Backend stores and returns only that user's messages.
8. Frontend refreshes chat every 2 seconds.

Expected behavior:
1. User A sees only User A messages.
2. User B sees only User B messages.
3. Message list persists in Supabase table.

## 10. API Endpoints

### `GET /messages`
- Returns authenticated user's messages.

### `POST /messages`
- Request body:

```json
{
	"username": "YourName",
	"text": "Hello"
}
```

- Stores message for authenticated user only.

Required header for protected APIs:

```http
Authorization: Bearer <supabase_access_token>
```

## 12. Common Issues and Fixes

### 1) 500 FUNCTION_INVOCATION_FAILED on Vercel
- Cause: Missing Vercel env vars.
- Fix: Add `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` in Vercel settings and redeploy.

### 2) `az` command not recognized
- Cause: Azure CLI path missing in terminal.
- Fix: Add Azure CLI install path to PATH and reopen terminal.

### 3) Login works but chat does not load
- Check `API_URL` in `frontend/script.js`.
- Check backend deployment URL is active.
- Check browser console/network for `/messages` errors.

Extra checks:
1. Confirm Vercel env vars are available in Production scope.
2. Confirm token is being sent in Authorization header.
3. Confirm RLS policies were created successfully.

### 4) No verification email received
- Check spam/junk folder.
- Confirm email provider settings in Supabase.
- Confirm user exists in Supabase Auth -> Users.

### 5) SQL failed in terminal
- SQL should be run in Supabase SQL Editor, not PowerShell/CMD.
- Copy SQL into Supabase dashboard query editor and run there.

## 13. Security Notes

- Never expose `SUPABASE_SERVICE_ROLE_KEY` in frontend code.
- Keep service role key only in backend environment variables.
- Rotate keys if they are accidentally shared publicly.

If any key is exposed publicly:
1. Rotate key immediately in Supabase.
2. Update backend env vars.
3. Redeploy backend.

## 15. Process Summary (For Beginners)

If you are doing this project from scratch, follow this exact order:

1. Build frontend UI with HTML/CSS/JS.
2. Build backend API with Express.
3. Create Supabase project and enable email auth.
4. Create messages table + RLS policies in Supabase SQL Editor.
5. Connect frontend to Supabase auth and backend API.
6. Deploy backend to Vercel and set backend env vars.
7. Update frontend API URL with deployed backend URL.
8. Deploy frontend to Azure Static Web Apps.
9. Test signup -> verify email -> login -> send message -> auto refresh.

That gives a complete cloud-deployed private one-way messenger.
